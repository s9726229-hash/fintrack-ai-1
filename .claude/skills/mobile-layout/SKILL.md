# FinTrackAI 手機版排版調整

這份 skill 記錄 2026-07-21 那次全站性手機版間距調整（[Layout.tsx](../../../components/Layout.tsx)、`TransactionFilters.tsx`、`Dashboard.tsx` 的 `CollapsibleSection` 等）之後定下來的規則，以及我們處理這類問題時的合作方式。目的是讓下一次調整不用重新診斷一輪，也不會做出跟上次矛盾的決定。

## 合作方式

使用者習慣先給手機截圖，指出哪裡「太擠、留白太多、被截斷、對不齊」這類觀感問題，而不是直接說要改哪個 class。處理順序：

1. **先找共同根因，不要逐頁各自修**。這個專案好幾個排版問題常常源自同一個結構性原因（例如上次的「雙重 padding」：`Layout.tsx` 外層跟各 view 自己都加了一層 `p-2 md:p-6`）。看到多個症狀時，先假設它們可能同源，讀程式碼確認根因，而不是每個症狀各開一個獨立 fix。
2. **診斷完先講清楚「為什麼」再動手**。依照使用者全域 CLAUDE.md 的開發流程，改版面前要先提出診斷＋修正方向，等使用者確認（尤其這是「一次全站性調整」這種影響範圍大的變更），才動手改。使用者可能會直接說「一次全站性調整」表示同意全部一起做，不需要每一項再問一次。
3. **改完一定要用真實資料在手機視窗實測**，不能只憑讀 code 判斷有沒有修好。方法見下方「驗證方式」。
4. **每次調整完，把新確立或修正的規則補進這份文件**（尤其是使用者事後指出「這樣還是不對」的地方），讓這份 skill 持續反映目前的實際規範，而不是只停在 2026-07-21 那次的快照。

## 已知結構與既有規則（2026-07-21 建立）

### Padding 只能有一層
`components/Layout.tsx` 的可捲動內容容器（`<div className="flex-1 overflow-y-auto ...">`）是**唯一**負責頁面邊距與底部安全區的地方：手機版 `px-3 pt-3 pb-24`、桌面版 `md:p-8 md:pb-8`。

各 `views/*.tsx` 的根 `<div>` **不要**在手機版加自己的 `p-2`／`pb-24`——之前每個 view 都各自加了一層，導致手機版邊距變成雙倍（24px 而非 12px），捲到底部也是雙倍留白。view 只需要 `md:p-6`（桌面版在 Layout 的 `md:p-8` 之外再多一層內縮，這是刻意的桌面版視覺，維持不動）。

新增 view 或還原被改壞的版面時，根 `<div>` 的 class 應該長這樣：
```
className="space-y-6 animate-fade-in md:p-6"
```
不要恢復成 `p-2 md:p-6 pb-24`。

### 標籤/數值不要用同一行擠壓，改成上下疊
凡是「短標籤 + 可能很長的數值」放在同一個 `justify-between` 橫向 row 裡、標籤那邊用 `truncate`、數值那邊用 `shrink-0` 的寫法，數值一長就會把標籤擠到被裁切成「...」（發生過在 Dashboard 的「總資產」「總負債」卡片）。

規則：短標籤＋數值的組合，**改成標籤在上、數值在下的兩行**，不要橫向搶位置。這樣無論數值多長，標籤永遠完整顯示。

### 金額類小卡數字要防止換行
KPI 小卡（尤其 3 欄並排的 grid，如股票投資頁的庫存總市值/未實現總損益/總報酬率）在窄螢幕下，數字容易被擠到換行變成兩行（例如「-$38.9」和「萬」分兩行）。

規則：
- 數值 `<div>` 加 `whitespace-nowrap`。
- 手機版字級不要無腦套 `text-base`（16px），3 欄並排這種寬度吃緊的情況用 `text-[15px]` 之類更保守的尺寸。
- 卡片 padding 在手機版也可以比預設再收緊一點（例如 `p-2.5` 而非 `p-3`）幫數值多爭取寬度。

### 輔助資訊（如日期區間）不要跟主要 tabs 擠同一行
`components/transactions/TransactionFilters.tsx` 曾經把時間篩選 tabs（月/季/半年/...）跟右側的日期區間徽章放在同一個 `flex flex-wrap` row、用 `ml-auto` 頂到最右邊。手機版 tabs 已經佔滿整行，徽章被迫換行變成自己獨佔一行、卻只用了右邊一小塊，整行左邊都在浪費。

規則：主要操作（tabs、按鈕列）跟輔助說明文字（日期區間、統計摘要之類）**各自獨立成一行**，不要靠 flex-wrap 讓它們共用又互相搶位置。輔助文字那一行可以就用純文字＋小圖示，不用刻意包一個徽章底色框，除非真的需要視覺強調。

### `CollapsibleSection`（Dashboard.tsx）標題列要有手機斷點
`CollapsibleSection` 元件（Dashboard.tsx 裡定義，用在「本月財務摘要」「總資產與負債趨勢」「預算執行度」等收合卡）的標題列 padding 是 `px-4 py-3 md:px-6 md:py-4`，展開內容是 `px-4 pb-4 md:px-6 md:pb-6`。**不要**改回單一固定值（例如只寫 `px-6 py-4`），那樣手機版標題列會顯得比實際內容（一行標題+箭頭）大一圈。

### 行內輸入框用 `flex-1`，不要寫死窄寬度
Dashboard 首屏的「今日尚未記帳」快速記帳輸入列，項目輸入框原本寫死 `w-16`（64px），在窄螢幕上明顯太小。改成 `flex-1`（撐滿可用剩餘寬度），金額框給固定但合理的寬度（`w-20`）。

規則：手機版行內表單如果要塞好幾個元素在一行，用 `flex-1` 讓文字類輸入框吃可用空間，不要為了「看起來整齊」而寫死一個很窄的 px/rem 寬度。

### Mobile Header 邊距要跟內容對齊
`components/layout/MobileHeader.tsx` 的左右 padding（`px-4`）跟高度（`h-14`）要跟 Layout 內容區的邊距風格一致，不要各自維護不同的邊距數字（之前 header 是 `px-6 h-16`，比內容區的邊距寬，看起來不對齊）。

## 頁面標準結構（2026-07-21 以股票投資頁為範本建立的 6 層架構）

使用者以「股票投資」頁為範例，定義了所有頁面由上到下該有的區塊順序，目的是讓全站頁面風格一致。之後新增頁面或調整既有頁面，先比對這個順序，缺哪一層或順序不對就是要修的地方：

1. **全域頂部列**：左＝App 名稱，右＝全域功能（配色切換/版本紀錄/系統設定）。對應 `components/layout/MobileHeader.tsx`（手機）與 `components/layout/Sidebar.tsx`（桌面），這層不用每頁重做。
2. **頁面標題列**：左＝頁面名稱＋icon＋副標題，右＝該頁全域功能鍵（例如匯入檔案、新增項目）。沒有對應功能就留空，不用硬塞。**主要操作按鈕一律用 `components/ui.tsx` 的共用 `Button` 元件**（`theme="warm"`），不要寫原生 `<button>` 另外刻樣式——之前 Assets、Recurring、Transactions 的「新增」按鈕都各自寫了一份原生 `<button>`，跟其他頁視覺細節（圓角/陰影/按壓效果）對不齊，已陸續修正為共用 `Button`。
3. **分頁列**：左＝分頁選項，右＝該分頁專屬的功能鍵（例如 Investments 庫存總覽的「更新全部現價」、股息分析的「AI 分析股息」）。

   **分頁的定義是「任何切換後會改變下方資料內容的按鈕組」，不是只有「切換整組資料集」這種才算**——使用者 2026-07-21 明確定義：股票投資的庫存/交易/股息切換是分頁、資產管理的資產類別篩選是分頁、收支記帳的月/季/半年/年度/全部/自訂義時間範圍切換也是分頁。只要符合這個定義，一律放在頁面標題列正下方（第3層），不要因為它實際功能是「篩選」還是「換頁」就放到不同層級或用不同樣式。

   **全部共用 `components/ui.tsx` 的 `Tabs` 元件**（底線樣式：`px-1 py-3 text-sm border-b-2`，選中＝`text-[#C4523A] border-[#C4523A] font-bold`），不要各頁自己刻一份 tab 樣式再各自微調間距字級——之前 Investments/Assets/Transactions 三處各自寫的 tab 樣式其實只差在 `gap-4` vs `gap-5`、`py-2` vs `py-3`、`text-xs` vs `text-sm` 這種細節，統一抽成 `Tabs` 元件後就不會再各自漂移。`Tabs` 接受 `options`/`active`/`onChange`/`rightSlot`（分頁專屬功能鍵放這裡）。

   **目前套用這個分頁定義與 `Tabs` 元件的地方**：
   - `Investments.tsx` 頁面級分頁（庫存總覽/交易紀錄/股息分析）。
   - `Assets.tsx` 頁面級分頁（全部/股票基金/現金存款/負債貸款）——2026-07-21 從 `AssetList.tsx` 內部拉到 `Assets.tsx` 標題列正下方，之前巢狀藏在列表元件裡、比標題列矮一層，已修正對齊。
   - `Transactions.tsx` 頁面級分頁（月/季/半年/年度/全部/自訂義）——2026-07-21 從「篩選」重新歸類成「分頁」，從 KPI 小卡下方移到標題列正下方。原本捆在一起的 `TransactionFilters` 元件拆成兩個：`TimeRangeTabs`（分頁本身＋選「自訂義」時緊接的起訖日期輸入框，因為那是操作這個分頁選項需要的參數）放在第3層；`TransactionFilters`（搜尋框＋目前時間範圍的實際日期區間文字）留在原本第5層篩選的位置，因為那是在已選定範圍內再細篩，跟分頁本身是不同層級的操作。
   - `Investments.tsx` 的「交易紀錄」分頁內部，也重用同一個 `TimeRangeTabs`（透過 `TransactionAnalysisView` 的 `timeRangeSlot` prop），放在該分頁內容最上方、KPI 小卡之前——因為頁面級分頁已經佔用第3層，巢狀的次級分頁就放在該分頁內容的最頂端，等同該子頁面自己的「第3層」。

4. **KPI／統計小卡**：卡片標題壓縮扁平化，卡片內留白比例抓在 20% 上下，不要為了「好看」硬塞大留白。數字類卡片要注意換行問題，見上面「金額類小卡數字要防止換行」那條規則。如果同一組資料項目很多（例如超過 5 個統計值），優先考慮一個長方形整合卡（單一 row 裡分欄顯示），而不是硬塞更多小方塊擠爆版面。分析型／洞察型卡片（例如 Recurring 的「財務狀況健康診斷」、Dashboard 的「本月財務摘要」）算這一層的延伸，不需要另外開一個新的位置。
5. **篩選功能**（如果有）：緊接在 KPI 卡之後、資料列表之前，是「在已選定的分頁範圍內再細篩」的功能（例如搜尋框、實際日期區間顯示），跟第3層的分頁不同層級——分頁本身（連同操作分頁需要的參數，如自訂義的起訖日）留在第3層，不要混進這裡。之前 Investments 的「交易紀錄」分頁把整組 `TransactionFilters`（含分頁+篩選）放在 KPI 卡「之前」，順序反了，已改成把分頁跟篩選拆開、各自歸位到第3層跟第5層，並透過 `TransactionAnalysisView` 的 `timeRangeSlot`／`filterSlot` 兩個 prop 分開插入——之後這類「卡片+列表包在同一個元件裡、分頁/篩選卻要插在中間」的情境，優先用這種 slot/prop 傳入的方式解決，不要為了塞內容硬把元件拆開重組。
6. **資料列表**：放在最下面，且**捲動時欄位表頭要凍結**（sticky）。目前的做法：桌面版用 `<table><thead className="sticky top-0">`；手機版沒有 `<table>` 可用（改用緊湊卡片列），要另外加一條 `md:hidden sticky top-0 z-10` 的欄位標籤列（欄位名稱可以精簡，不用跟桌機逐欄對應），放在捲動內容的最前面，讓它跟著同一個 `overflow-y-auto` 容器固定在頂端。已套用這個模式的地方：`AssetList.tsx`、`Investments.tsx`（庫存明細）、`TransactionHistoryList.tsx`；`TransactionList.tsx`（收支記帳）原本就是這個模式的範本，日後可以直接參考它的寫法（sticky 標題列 + sticky 欄位列，兩層都 `sticky`、`z-index` 依序疊高）。

   **單筆資料列（row）內部的左右順序，固定是「圖案／名稱／數據／操作」四段**，由左到右：
   - **最左＝圖案**：代表該筆資料類別的圖示或色條（例如資產類型的顏色色條、交易收支的 icon、股票買賣的 BUY/SELL 徽章）。這段一定要有，不能省略，是使用者辨識列別最快的視覺線索。
   - **接著＝名稱**：主體名稱（資產名稱、股票名稱、交易項目），可以帶次要說明文字（代號、分類、日期）在名稱下方一起顯示，這段用 `flex-1 min-w-0` 讓它吃掉可用寬度。
   - **再來＝數據**：金額／數值，靠右對齊、`shrink-0`、`whitespace-nowrap`，需要的話下面再帶一行次要數據（漲跌%、損益）。
   - **最右＝操作**：編輯/刪除/更新等操作按鈕，永遠放最後、`shrink-0`。

   **目前全部符合這個順序**：`TransactionList.tsx`（收支記帳，圖案用 `getIcon(t.type)`）、`AssetList.tsx`（資產管理，圖案用色條 `warmColor.bar`）、`TransactionHistoryList.tsx`（股票交易紀錄，圖案用買/賣徽章）、`Investments.tsx` 庫存明細手機版列表（2026-07-21 補上，圖案用當筆損益顏色的色條：賺=紅 `#C4523A`／賠=綠 `#6B9080`／持平=灰 `#A69B87`，跟這格右側「數據」欄本來就有的損益配色呼應，不是另外發明一套顏色系統）。

### 已知例外／使用者確認過的取捨（2026-07-21）
- **Dashboard（總覽儀表板）暫不套用分頁統一化，先維持原樣**：它內部其實也有兩組符合「切換後改變下方資料」定義的分頁（圖表類型切換、債務卡的還款總覽/寬限與跑道/利息成本/扣款帳戶/提前還款模擬），但目前用的是圓角按鈕組樣式，不是 `Tabs` 元件的底線樣式，跟其他頁不一致。使用者 2026-07-21 明確表示「總覽先不要動，先把其他分頁格式全數固定」——這是刻意延後處理，不是判斷 Dashboard 不需要統一，之後如果要動 Dashboard，記得把這兩組也換成 `Tabs` 元件。標題列的部分已經補上（見 `views/Dashboard.tsx` 最上方新增的標題列），其餘維持原有的堆疊結構。
- **Recurring 的項目列表維持卡片格狀排列**，不強改成有 sticky 表頭的捲動列表——固定收支項目通常數量少（訂閱/房租/貸款），格狀卡片比清單更適合這種內容量，沒有「資料太多需要凍結表頭」的實際需求。
- **Budget/Guide/Settings 標題列右側留空**：這幾頁本來就沒有對應的全域功能（沒有匯入/新增這類動作），留空是正確的，不用硬塞一個功能鍵進去湊版面一致。

## 驗證方式（改完排版一定要做）

1. 用 `mcp__Claude_Browser__preview_start` 啟動 `FinTrack AI Dev Server`（`.claude/launch.json` 裡的設定，port 3000）。
2. 用 `mcp__Claude_Browser__resize_window` 切到 `preset: "mobile"`（375x812）。
3. **用使用者真實備份資料**灌進 localStorage，不要只看空狀態或隨便造的假資料——排版問題（文字長度、數字位數、換行）往往只有真實資料量才會暴露出來。做法見 [[project-fintrackai-dividend-verification]]：
   - 最新備份在 `Desktop/財務管理/fintrack_ai_backup_*.json`，先複製到 scratchpad，用 node 腳本移除 `ft_api_key`/`ft_finmind_token`/`ft_google_client_id` 三把金鑰。
   - 把去敏後的 JSON 複製到專案 `public/` 資料夾（暫時的，測完要刪），在瀏覽器裡 `fetch` 它，逐一 `localStorage.setItem(key, JSON.stringify(value))` 寫入（key 對照 `constants.ts` 的 `STORAGE_KEYS`），然後 reload。
   - 測完務必刪掉 `public/` 下的暫存 JSON、確認 `git status` 乾淨。
4. 用 `read_page`（`filter: interactive`）配合 `computer` 點擊切換到要檢查的分頁，用 `computer { action: "screenshot" }` 或 `get_page_text` 確認實際渲染結果，逐頁比對修正前後的差異。
5. 改完也要用 `resize_window preset: "desktop"` screenshot 一次，確認桌面版沒有被手機版的調整波及（這次全站調整刻意只動 mobile-only 的 class，`md:` 開頭的桌面樣式不動）。
6. **已知工具怪癖**：桌面尺寸（尤其手動指定 1280x900 這類寬視窗）偶爾會出現 screenshot 截圖把畫面內容壓縮擠在左上角一小塊、其餘留白的情況，但這通常只是截圖工具本身的顯示問題，不是真的排版壞了——用 `javascript_tool` 檢查 `window.innerWidth`／`document.querySelector('main').getBoundingClientRect()` 確認實際 DOM 尺寸是否正常，或改用 `resize_window` 指定 1024x768 這類較保守的寬度通常畫面就會正常。同時 `computer` 的 `ref` 點擊在這個怪異狀態下可能也會對不準（點擊座標跟畫面對不上），此時改用 `coordinate` 直接點畫面上看到的位置，或縮小視窗寬度後再重新 `read_page` 取得新的 ref。遇到這狀況不用懷疑是自己剛剛改的 code 造成的，先用 DOM 尺寸／`get_page_text` 確認內容正確性即可。
7. **`computer` 點擊有時完全沒反應（連在 mobile 375px 視窗都會發生）**：`read_page` 抓到的 ref 看起來正常、`elementFromPoint` 也確認座標對到正確的按鈕，但 `computer left_click` 點下去畫面就是沒切換，且用 screenshot 檢查同時會看到上一條「內容擠在左上角」的怪癖同時出現——這兩個怪癖看起來會同時發作。這種情況不用一直重試 `computer` 點擊，直接用 `javascript_tool` 執行 `document.elementFromPoint(x, y).closest('button').click()` 讓瀏覽器原生觸發那顆按鈕的 click，這樣一定會動作到，之後再用 `get_page_text` 或重新 screenshot 確認畫面已經正確切換（通常這麼做一次之後，screenshot 顯示也會跟著恢復正常）。

## 待補充

目前規則都是從 2026-07-21 那輪全站調整歸納出來的，還沒涵蓋所有版面情境（例如 modal/dialog 手機版、圖表容器手機版寬度）。使用者提到之後會不定期預擬新的排版構想來討論——每次確立新規則或推翻舊假設，都要回來更新這份文件，讓它保持跟現況同步，而不是只停留在第一版快照。
